# 02_FEATURE_SPEC：功能细化

## 1. 用户认证模块

### 功能列表

- 邮箱验证码注册；
- 邮箱密码登录；
- 邮箱验证码找回密码；
- JWT 令牌签发；
- 获取当前用户信息；
- 用户退出登录；
- 管理员登录；
- 基于角色的接口权限控制。

### 用户角色

```text
USER  普通用户
ADMIN 管理员
```

所有可登录账号统一存储在 `users` 表，使用 `role` 字段区分角色。管理员不开放注册，由 Flyway 初始化或后台创建。

登录入口必须隔离：

- 用户端登录只允许 `USER`；
- 管理后台登录只允许 `ADMIN`；
- 普通用户不能登录管理后台；
- 管理员不能登录用户端。

### 注册流程

```text
用户输入邮箱
↓
点击发送验证码
↓
后端生成 6 位验证码并写入 Redis
↓
邮件服务发送验证码
↓
用户填写验证码、昵称、密码和确认密码
↓
后端校验验证码和密码强度
↓
使用 BCrypt 存储密码哈希
↓
创建用户账号
```

### 登录流程

```text
用户输入邮箱和密码
↓
后端校验账号状态
↓
使用 BCrypt 校验密码
↓
签发 JWT accessToken
↓
前端保存 Token 并进入系统
```

### 找回密码流程

```text
用户输入邮箱并发送验证码
↓
后端校验邮箱是否存在
↓
用户提交验证码和新密码
↓
后端校验验证码
↓
更新密码哈希
↓
清理验证码并要求重新登录
```

### 验证码限制

- 验证码默认 5 分钟有效；
- 同一邮箱 60 秒内不能重复发送；
- 同一邮箱每日最多发送 10 次；
- 同一 IP 每小时最多发送 20 次；
- 验证码错误 5 次后失效。

## 5. 订票模块

### 订票流程

```text
查询航班
↓
查看详情
↓
选择乘机人
↓
选择座位
↓
提交订单
↓
模拟支付
↓
出票成功
```

### 座位状态

```text
AVAILABLE 可选
LOCKED    锁定中
SOLD      已售
DISABLED  不可选
```

### 座位锁定

用户选择座位提交订单时，系统将座位从 `AVAILABLE` 更新为 `LOCKED`。

支付成功后，座位更新为 `SOLD`。

如果订单超时未支付，座位释放回 `AVAILABLE`。

### 乘机人与座位映射

创建订单时，乘机人与座位必须一一对应。后端不得只依赖两个数组的顺序隐式匹配，推荐请求结构使用：

```json
{
  "flightId": 1001,
  "items": [
    {
      "passengerId": 1,
      "seatId": 10001
    }
  ]
}
```

校验规则：

- `items` 不能为空；
- 同一订单内 `passengerId` 不允许重复；
- 同一订单内 `seatId` 不允许重复；
- 所有乘机人必须属于当前登录用户；
- 所有座位必须属于同一个目标航班；
- 座位状态必须为 `AVAILABLE`；
- 儿童和婴儿乘客如进入实现范围，必须在订单明细中记录关联成人乘客。

## 6. 并发库存控制模块

核心目标：防止多个用户同时购买同一个座位。

推荐 SQL：

```sql
UPDATE flight_seat
SET status = 'LOCKED',
    version = version + 1,
    locked_by_order_id = #{orderId},
    lock_expire_time = #{lockExpireTime}
WHERE id = #{seatId}
  AND status = 'AVAILABLE'
  AND version = #{version};
```

如果影响行数为 1，则锁座成功；如果为 0，则表示座位已被其他用户占用。

## 7. 订单模块

- 查询订单列表；

## 8. 候补模块

当航班目标舱位无可售座位时，用户可以提交候补订单。候补采用预付式设计：用户提交候补单时先完成支付，候补成功后系统直接出票；候补失败、用户取消或航班起飞前仍未兑现时，系统按原支付金额退回。

候补订单需要记录候补乘机人明细。单人候补和多人候补都通过 `waitlist_passenger` 保存乘机人快照，候补兑现成功后再把分配座位写回明细。

### 候补下单流程

```text
用户选择候补航班、目标舱位和乘机人
↓
提交候补订单
↓
生成待支付候补单
↓
模拟支付候补金额
↓
候补单进入排队中
```

候补支付金额按提交候补时的目标舱位价格和乘机人数计算，并写入候补单。支付成功后，候补单进入正式排队，不再要求用户二次支付。

### 候补状态

```text
PENDING_PAYMENT 待支付
WAITING         已支付，排队中
SUCCESS         候补成功，已出票
CANCELLED       用户已取消
FAILED          候补失败
REFUNDED        已退款
EXPIRED         支付超时
```

候补单在支付成功后从 `PENDING_PAYMENT` 进入 `WAITING`。`WAITING` 状态表示用户已付款并正在等待座位释放，不再存在锁座后待支付状态。

### 触发时机

当用户退票成功后：

```text
释放座位
↓
查询最早的已支付候补订单
↓
为候补订单分配座位
↓
创建正式票务订单并出票
↓
候补状态改为 SUCCESS
```

### 候补分配规则

候补只处理 `WAITING` 状态的已支付候补单。处理顺序按 `paid_at` 升序，时间相同按 `id` 升序，保证稳定的先付款先服务。候补兑现只能分配候补单目标舱位内的座位。

多人候补必须整体满足，不做部分出票。例如当前释放 1 个座位，而队首候补需要 2 个座位，则该候补继续等待，系统可以继续检查后续只需要 1 个座位的候补；该策略需要在候补记录中保留跳过原因，便于后台排查。

候补兑现成功时，系统需要在同一事务中完成：

- 将分配座位更新为 `SOLD`；
- 创建状态为 `ISSUED` 的正式 `ticket_order` 和 `order_passenger` 快照；
- 把正式订单 ID 写回 `waitlist_order.ticket_order_id`；
- 把分配座位写回 `waitlist_passenger.locked_seat_id` 和 `locked_seat_no`；
- 候补状态改为 `SUCCESS`。

`SUCCESS` 候补必须能够跳转到对应订单详情。

### 支付与退款规则

如果候补单在 `PENDING_PAYMENT` 状态下超时未支付：

- 候补状态改为 `EXPIRED`；
- 不进入候补队列；
- 不占用座位。

用户主动取消 `WAITING` 候补时：

- 候补状态改为 `CANCELLED`；
- 发起原路退款；
- 退款成功后状态改为 `REFUNDED`。

如果航班起飞前候补仍未兑现，或航班取消导致候补无法继续处理：

- 候补状态改为 `FAILED`；
- 发起原路退款；
- 退款成功后状态改为 `REFUNDED`；
- 不创建正式票务订单。

候补退款结果记录在候补单中，便于用户在候补详情和后台订单管理中查询；候补成功后再退正式订单时，按普通退票流程生成退票记录。

## 9. 退票模块

退票规则示例：

- 起飞前 24 小时以上：手续费 10%；
- 起飞前 2 到 24 小时：手续费 30%；
- 起飞前 2 小时内：不可退票。

退票成功后：

- 订单状态改为 `REFUNDED`；
- 座位状态改为 `AVAILABLE`；
- 生成退票记录；
- 触发候补处理。

## 10. 改签模块

改签流程：

```text
选择原订单
↓
查询可改签航班
↓
选择新航班和新座位
↓
计算差价与手续费
↓
确认改签
↓
释放旧座位，锁定新座位
↓
订单状态更新
```

基础规则：

- 只有 `ISSUED` 订单允许改签；
- 新航班必须处于可售状态；
- 新座位必须为 `AVAILABLE`；
- 改签确认必须在同一事务中完成旧座位释放、新座位锁定、差价记录和订单状态更新；
- 任一环节失败必须整体回滚；
- 加分版本可以把改签做成完整流程，MVP 和推荐完整版本可以只保留接口设计和状态定义。

### 航班管理

- 新增航班；
- 编辑航班；
- 上架航班；
- 下架航班；
- 生成座位。

### 订单管理

- 查询订单；
- 查看订单详情；
- 代指定普通用户创建订单；
- 管理员发起退票或改签；
- 作废已取消或已退票订单；
- 更新管理员备注；
- 查看退改签记录。

管理员代下单必须复用普通下单的锁座、金额和乘客快照规则，订单归属为目标 `userId`，创建后保持 `PENDING_PAYMENT`，仍由用户端支付出票。管理员退票和改签必须复用普通流程的座位释放、余票维护、候补兑现、费用计算和改签记录；`force=true` 只允许跳过距起飞不足 2 小时的窗口限制，费用按不足 24 小时档 30% 计算。

订单作废是逻辑状态变更，仅允许 `CANCELLED` 或 `REFUNDED` 进入 `VOIDED`。作废本身不释放座位、不回补余票、不改支付或退改签记录。订单“修改”能力只开放 `admin_note` 备注字段，金额、航班、乘机人和座位等结构化变化必须通过退票、改签或作废完成。

### 普通用户管理

- 查询普通用户；
- 新增普通用户；
- 禁用普通用户；
- 恢复普通用户；
- 逻辑删除普通用户。

该模块只面向 `role = USER` 的普通用户账号，不用于管理 `role = ADMIN` 的管理员账号。管理员新增用户时固定创建 `role = USER`、`status = NORMAL`、`email_verified = 0` 的账号，密码使用 BCrypt 入库，不触发验证码或验证邮件。

禁用和删除前必须检查目标用户是否存在未完成订单、进行中候补、处理中退票或改签流程；存在则拒绝。逻辑删除通过 `users.status = DELETED` 表示，账号行和历史业务数据保留。`DELETED` 是终态，不能再次禁用、删除或启用；启用只允许 `DISABLED -> NORMAL`，防止已删除账号复活。`DISABLED` 与 `DELETED` 用户都不能登录用户端。

本次订单和用户管理端写操作必须写入 `admin_operation_log` 审计，包括代下单、管理员退票、管理员改签、作废、备注更新、用户新增、删除、禁用和启用；既有航班、座位和 AI LLM 配置写操作不纳入本次新增审计范围。
