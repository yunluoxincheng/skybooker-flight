# 01_REQUIREMENTS：需求分析

## 0. 项目名称

- 中文名：云航智订
- 英文名：SkyBooker
- 仓库名：`skybooker-flight-system`
- 项目副标题：基于 Next.js + Spring Boot 的智能航班查询与机票预订系统

## 1. 项目目标

本项目目标是设计并开发一套航班查询与机票预订业务系统，解决传统售票方式效率低、信息不透明、用户体验差的问题，为用户提供在线查询、筛选、预订、选座、退改签和智能购票推荐能力。

项目需要满足 Java Web 实训要求，同时按照真实项目方式进行工程化设计。

## 2. 用户角色

### 普通用户

普通用户可以注册登录、查询航班、筛选航班、查看航班详情、选择乘机人、选座、提交订单、模拟支付、查看订单、退票、改签、提交候补订单以及使用 AI 智能购票助手。

### 管理员

管理员可以维护航班、航司、机场、座位、订单和普通用户，并查看数据统计报表。

管理员账号不开放注册，由系统初始化或后台创建。所有登录账号统一存储在 `users` 表中，通过 `role = ADMIN` 判断后台权限。管理后台入口与用户端登录入口隔离，普通用户不能登录管理后台，管理员也不能登录用户端。

## 3. 功能性需求

### 航班查询

系统需要支持两种基础查询方式：

1. 给定日期 + 航班号；
2. 给定日期 + 出发地 + 目的地。

查询结果至少包含：

- 航班号；
- 航空公司；
- 出发地与目的地；
- 出发机场与到达机场；
- 起飞时间与到达时间；
- 飞行时长；
- 剩余座位数；
- 票价；
- 航班状态。

高级筛选包括：

- 航空公司；
- 起降时间；
- 飞行时长；
- 价格区间；
- 直飞/中转；
- 行李额；
- 航班状态。

### 机票预订

系统需要支持：

- 对指定日期和指定航班进行机票预订；
- 选择乘机人；
- 可视化选座；
- 模拟支付；
- 正确扣减座位库存；
- 多人并发预订时避免超卖；
- 无票时可以提交候补订单并先完成支付。

### 退改签

系统需要支持：

- 用户在线退票；
- 根据退票规则计算手续费；
- 退票后释放座位；
- 若存在已支付候补订单，优先为候补用户兑现出票；
- 用户在线改签；
- 改签时计算差价和手续费。

### 管理后台

后台需要支持：

- 航班录入、更新、下架；
- 航司和机场管理；
- 用户账户管理；
- 订单管理；
- 退改签记录管理；
- 数据统计报表。

### AI 智能购票助手

AI 助手需要支持：

- 用户自然语言表达需求；
- 系统解析目的地、出发地、日期、时间偏好、价格偏好等字段；
- 如果缺少必要条件，主动追问；
- 如果条件完整，查询数据库中真实可购买航班；
- 推荐前三个符合条件的航班；
- 给出跳转链接，用户可点击进入航班筛选页、详情页或预订页。

## 4. 非功能性需求

### 可用性

- 页面布局清晰；
- 查询流程简单；
- 预订流程步骤明确；
- 管理后台操作高效。

### 一致性

- 所有订单状态必须明确；
- 座位状态和订单状态必须保持一致；
- 退票后必须释放座位；
- 候补出票必须遵循先来先服务原则。

### 并发安全

- 多用户同时抢同一座位时，最多只能成功一个；
- 座位库存扣减必须在事务中完成；
- 使用乐观锁或数据库条件更新防止超卖。

### 可扩展性

- AI 助手默认使用规则解析，可扩展为 LLM 解析；
- 订单、退票、改签、候补模块解耦；
- 前端按 features 模块拆分，便于多人开发。

## 5. 交付范围与验收标准

为避免范围失控，本项目按三个层级验收。

### 5.1 MVP 必须完成

- 系统可以正常启动；
- Flyway 可以自动初始化数据库；
- 普通用户可以注册、登录和退出；
- 管理员可以通过后台入口登录；
- 用户可以查询航班、查看航班详情和座位图；
- 用户可以维护乘机人；
- 用户可以选择乘机人和座位，创建订单并完成模拟支付；
- 多用户同时抢同一座位时，最多只有一个订单创建成功；
- 用户可以查看我的订单；
- 管理员可以维护航班并查看订单；
- 视频演示和 PPT 至少覆盖登录、航班查询、选座下单、模拟支付、订单查看和后台航班管理。

### 5.2 推荐完整版本

在 MVP 基础上增加：

- 高级筛选和排序；
- 退票及手续费计算；
- 退票释放座位；
- 候补下单支付、排队兑现和失败退款；
- AI 助手根据自然语言推荐真实可购买航班；
- 管理后台数据统计；
- Docker Compose 部署。

### 5.3 加分版本

在完整版本基础上增加：

- 改签完整流程；
- LLM 意图解析；
- JMeter 并发测试报告；
- 服务器部署；
- 更完整的数据报表。

### 用户认证

系统需要提供完整的用户认证能力：

- 用户可使用邮箱验证码注册账号；
- 用户可使用邮箱和密码登录系统；
- 用户忘记密码时，可通过邮箱验证码重置密码；
- 系统使用 JWT 管理登录态；
- 密码必须使用 BCrypt 等安全哈希算法存储，不允许明文保存；
- 邮箱验证码存储在 Redis 中，默认有效期 5 分钟；
- 系统应限制验证码发送频率，防止恶意刷验证码；
- 管理员账号不开放注册，由系统初始化或后台创建；
- 手机验证码、微信登录、支付宝登录作为后续扩展，不作为实训主流程。
